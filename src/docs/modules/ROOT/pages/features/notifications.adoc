= Notifications
Paul Snow

:description: Guide to workout reminder and streak notifications in RepFoundry.

RepFoundry can send scheduled push notifications to remind users to work out on chosen days and to protect active training streaks.

== Overview

Notifications are delivered via the `flutter_local_notifications` package using timezone-aware scheduling from the `timezone` package.
`NotificationService` is a singleton that manages all scheduling.
User preferences are stored in `ReminderSettings` and persisted via SharedPreferences through `ReminderSettingsNotifier`.

== ReminderSettings Model

[source,dart]
----
class ReminderSettings {
  final Set<int> enabledDays;        // DateTime.monday (1) through DateTime.sunday (7)
  final int hour;                    // 0–23; default 18
  final int minute;                  // 0–59; default 0
  final bool streakReminderEnabled;  // default false
}
----

The `hasReminders` computed property returns `true` when `enabledDays` is non-empty.

== NotificationService

`NotificationService` is a singleton accessible via its factory constructor (`NotificationService()`), and is injected into Riverpod consumers via `notificationServiceProvider` so tests can swap it for a fake.

=== Initialisation

[source,dart]
----
Future<void> init()
----

Must be called once during app startup (before any scheduling).
Initialises timezone data (`tzdata.initializeTimeZones()`) and configures the `FlutterLocalNotificationsPlugin`:

* Android — uses `@mipmap/ic_launcher` as the notification icon
* iOS — requests alert, badge, and sound permissions via `DarwinInitializationSettings`

=== Scheduling Weekly Reminders

[source,dart]
----
Future<void> scheduleWeeklyReminders(ReminderSettings reminderSettings)
----

Cancels all existing reminders, then schedules one notification per enabled day using `FlutterLocalNotificationsPlugin.zonedSchedule()`.

Notifications repeat weekly (`matchDateTimeComponents: DateTimeComponents.dayOfWeekAndTime`).

=== Scheduling the Streak Reminder

[source,dart]
----
Future<void> scheduleStreakReminder(int hour, int minute)
----

Schedules a single daily notification (notification ID `100`) that fires at the given time each day (`matchDateTimeComponents: DateTimeComponents.time`).
The notification message warns the user that they have not yet logged a workout today.

=== Cancellation

[source,dart]
----
Future<void> cancelStreakReminder()   // cancels notification ID 100
Future<void> cancelAllReminders()    // cancels IDs 1–7 and 100
----

Weekly reminder notifications use IDs 1–7 (one per day of week).

=== Permission API

[source,dart]
----
Future<NotificationPermission> permissionStatus()  // unknown | granted | denied
Future<bool> requestPermission()                  // returns granted state
Future<void> openNotificationSettings()           // launches system settings
Future<void> showTestNotification()               // fires an immediate notification (ID 999)
----

`NotificationPermissionNotifier` (an `AsyncNotifier`) wraps these calls so the UI can observe and react to permission changes.
The notifications screen calls `request()` lazily — only when the user enables their first reminder day, the streak toggle, or the test notification button.

== Scheduling Algorithm

The `_nextInstanceOfDayAndTime` helper finds the next calendar occurrence of the target `(weekday, hour, minute)` tuple in the device's local timezone.
If the computed time has already passed today, the schedule is advanced by 7 days.
The `_nextInstanceOfTime` helper performs the same calculation for the daily streak reminder without a weekday constraint.

== Android Permissions

Declare the following in `AndroidManifest.xml`:

[cols="1,2"]
|===
| Permission | Purpose

| `POST_NOTIFICATIONS`
| Required on Android 13+ (API 33) to show notifications

| `SCHEDULE_EXACT_ALARM` or `USE_EXACT_ALARM`
| Required for exact alarm scheduling; `inexactAllowWhileIdle` is used for battery efficiency but still requires the permission on some OEM configurations

| `RECEIVE_BOOT_COMPLETED`
| Re-schedules notifications after device restart
|===

NOTE: RepFoundry uses `AndroidScheduleMode.inexactAllowWhileIdle` for all reminders.
This balances battery life against delivery reliability; reminders may fire a few minutes late during Doze mode.

== iOS Permissions

iOS permissions (`alert`, `badge`, `sound`) are requested at initialisation time via `DarwinInitializationSettings`.
No additional `Info.plist` keys are required beyond those added by the `flutter_local_notifications` package.

== Settings UI

Notifications have a dedicated screen at the route `/settings/notifications`, reached from a *Notifications* tile in the Settings hub.
The tile's subtitle summarises current configuration (for example, `Mon, Wed, Fri at 18:00`) or shows _No reminders configured_ when none are enabled.

The notifications screen contains:

* *Permission denied banner* — appears at the top when notifications are blocked at the OS level, with an *Open settings* action that launches the system notification settings via the `app_settings` package
* *Day picker* — a row of toggle chips for Monday through Sunday
* *Time picker* — opens the platform time picker; updates `hour` and `minute`
* *Streak reminder toggle* — enables or disables the daily streak protection notification
* *Send test notification* — fires an immediate notification so the user can verify their setup is working; disabled when permission is denied

The first time the user enables a reminder day, the streak toggle, or taps the test notification button, the OS permission prompt appears.
If the user denies permission, the toggle still persists their choice but the banner appears so they can re-grant later.

Changes are saved immediately via `ReminderSettingsNotifier`, which reschedules all active notifications to reflect the updated preferences.
